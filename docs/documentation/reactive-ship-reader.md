# Реактивный чтец SHiP

**Реактивный чтец** — сервисный компонент экосистемы COOPOS: библиотека и примеры на TypeScript, которая подключается к узлу с **State History (SHiP)** по **WebSocket**, принимает **бинарный поток** блоков, трасс и дельт таблиц и превращает его в **реактивные потоки** данных для приложений.

Исходный код и документация по использованию: репозиторий [coopenomics/coopos-ship-reader](https://github.com/coopenomics/coopos-ship-reader).

---

## Зачем он нужен

Протокол SHiP на стороне `nodeos` **намеренно минимален**: узел отдаёт сырые бинарные сообщения, не тратя CPU на интерпретацию контрактов. **Клиент** обязан хранить актуальные **ABI**, десериализовать действия и строки таблиц и учитывать **микрофорки** (откаты ветки), если строите согласованное с цепочкой состояние вне узла.

Реактивный чтец как раз закрывает эту часть: подписка на данные оформлена через **RxJS**, тяжёлая десериализация может распределяться по **worker threads**, что упрощает построение индексаторов, синхронизации с **PostgreSQL**, GraphQL-слоёв и других сервисов, которым нужна **непрерывная история**, а не единичные HTTP-запросы.

---

## С чем работает в связке

| Компонент | Роль |
|-----------|------|
| Узел COOPOS с `state_history_plugin` | Источник потока по SHiP (TCP или Unix socket). См. [State History (SHiP)](blockchain/state-history-ship.md). |
| **State-нода** (выделенная или совмещённая с API) | Типовой хост SHiP для «архивной» подачи истории. |
| **Light-нода** | Содержит состояние с момента снимка и новые блоки; если на ней включён SHiP, чтец может получать поток **с доступного на узле диапазона блоков** (не полная история с генезиса). Часто индексатор подключают к отдельной полной state-ноде, а light используют для локального API и P2P. |
| Реактивный чтец | Подключение к `ws_url`, запрос диапазона блоков, обработка потоков `blocks$`, дельт строк, ABI. |

Транспорт и семантика сообщений соответствуют протоколу **SHiP** узла COOPOS.

---

## Как это устроено кратко

1. Приложение получает `get_info` с HTTP узла (например, тот же хост, что и Chain API), чтобы узнать актуальную голову цепи и при необходимости `chain_id`.
2. Создаётся конфигурация чтеца: адрес WebSocket SHiP (`state-history-endpoint` или эквивалент), число потоков десериализации, белые списки дельт/таблиц, параметры запроса (`start_block_num`, флаги `fetch_block` / `fetch_traces` / `fetch_deltas` и т.д.).
3. После `start()` клиент получает **Observable**-потоки: блоки, строки таблиц по дельтам, обновления ABI по мере `setabi` в цепи.
4. Подписчики (или операторы RxJS) записывают данные во внешнее хранилище, публикуют события в шину или обновляют кэш.

Подробные примеры кода — в каталоге `examples` и в README репозитория [coopos-ship-reader](https://github.com/coopenomics/coopos-ship-reader).
